Teste de Sistemas · S.A. Final em grupo

O delivery — organizar, ligar as rotas e testar

Vocês — um grupo de 3 pessoas — recebem o código de um delivery de comida em React: 25 componentes e 5 páginas que funcionam, mas o projeto chegou desorganizado, sem as rotas ligadas e sem nenhum teste. O trabalho segue quatro etapas, nesta ordem:

1
Rodar e diagnosticar — instalar, tentar rodar e entender o que está errado.
2
Fazer a loja montar — completar os import do App.jsx e ligar as rotas.
3
Organizar as pastas — src/components/ e src/pages/, com os imports consertados.
4
Proteger com testes — um .test.jsx para cada um dos 25 componentes, dividido entre os 3.

Entrega final: npm run dev mostrando a loja + npm test com Test Suites: 25 passed.

1

O que vocês receberam

O cenário

Uma equipe entregou às pressas o front-end de uma delivery em React. O código funciona — cada componente e cada página fazem o que devem — mas o projeto chegou em mau estado: os 30 arquivos estão soltos na raiz de src/, o App.jsx está sem os imports (a loja nem monta) e não existe um teste sequer. Os produtos vêm de um arquivo JSON, servidos pela camada src/api.js.

O grupo de vocês assume esse projeto. Dois aliados: cada arquivo começa com um comentário dizendo o que o componente faz, quais props recebe e de que tipo de teste ele é — leiam sempre antes de mexer. E nada aqui exige ferramenta nova: tudo foi ensinado no conteúdo de React Testing Library. Mantenham o material aberto ao lado.

A regra deste projeto

Nos testes, encontre cada elemento do jeito que o usuário o percebe: pelo papel (getByRole), pelo texto (getByText), pela etiqueta do campo (getByLabelText) ou pela descrição da imagem (getByAltText) — e prove o comportamento agindo como um usuário agiria. Nunca procure por classe CSS nem por id.

2

O que vocês entregam

O mesmo delivery, em três condições:

  1. Montando — npm run dev abre a loja no navegador e as 5 rotas navegam.
  2. Organizada — componentes em src/components/, páginas em src/pages/, imports corrigidos.
  3. Protegida — um teste automatizado para cada um dos 25 componentes, todos verdes no npm test.

São os três itens do checklist da seção 4. No caminho, vocês exercitam as duas capacidades da UC: preparar o ambiente de testes e registrar o resultado de cada teste.

3

Roteiro de trabalho

Quatro etapas, na ordem. As etapas 1 a 3 o grupo faz junto — todos precisam entender a estrutura do projeto. A etapa 4 é dividida entre as 3 pessoas. Cada etapa termina com um "deu certo se…" — só avancem quando ele passar.

01Juntos

Rodem o projeto e diagnostiquem o problema

Clonem o projeto do GitHub, entrem na pasta da versão do grupo e rodem, nesta ordem:

terminalbash
# 0) baixa o projeto e entra na pasta da SUA versão
git clone https://github.com/jhonatanwll/sa-final-teste-sistemas.git
cd sa-final-teste-sistemas/versoes/04-delivery

# 1) instala React, Vite, React Router, Jest, RTL e o json-server
npm install

# 2) (opcional) sobe a API de mentira que serve o db.json
npm run api      # http://localhost:3001/produtos

# 3) tenta subir a loja — repare no que acontece
npm run dev

# 4) roda os testes — repare no que ele responde
npm test

Vocês vão esbarrar em três problemas. Cada um é resolvido em uma etapa:

A
A loja não monta. O App.jsx usa o React Router, o cabeçalho, o rodapé e as 5 páginas — mas não importa nada disso: faltam os import no topo do arquivo. → Etapa 2.
B
Tudo amontoado. Os 25 componentes e as 5 páginas estão soltos na raiz de src/, sem separação — difícil de achar, manter e testar. → Etapa 3.
C
Zero testes. Não há um único arquivo .test.jsx protegendo os componentes. → Etapa 4.
Deu certo se…

…o grupo consegue abrir o App.jsx e apontar, no código, por que a loja não monta.

¶
A aplicação consome o JSON pela camada src/api.js: ela tenta a API (json-server) e, se não estiver no ar, usa o src/produtos.json local — a loja monta com ou sem servidor. Precisa de Node 20+.
02Juntos

Façam a loja montar

Completem os import que faltam no topo do App.jsx. O próprio arquivo mostra o que ele usa: o BrowserRouter, Routes e Route (que vêm do react-router-dom), o Cabecalho, o Rodape e as 5 páginas.

Atenção ao tipo de export de cada arquivo antes de escrever o import: as páginas usam export default e os componentes usam export nomeado — o jeito de importar muda conforme isso (veja o glossário).

Deu certo se…

…npm run dev mostra a loja montada e as 5 rotas navegam: / (home), /produto/:id, /carrinho, /contato — e um endereço inventado cai na página "não encontrada" (rota *).

03Juntos

Organizem as pastas

Movam os 25 componentes para src/components/ e as 5 páginas para src/pages/ — o padrão que toda equipe React usa:

Como chegou
src/
├─ App.jsx        ← sem os imports
├─ api.js
├─ produtos.json
├─ Cabecalho.jsx
├─ CardProduto.jsx
├─ …               (25 componentes)
├─ HomePage.jsx
└─ …               (5 páginas)
Como entregar
src/
├─ App.jsx        ← imports certos
├─ api.js
├─ produtos.json
├─ components/
│  ├─ Cabecalho.jsx
│  ├─ Cabecalho.test.jsx  ← etapa 4
│  └─ …           (os 25 + testes)
└─ pages/
   ├─ HomePage.jsx
   └─ …           (as 5 páginas)

Mover quebra os caminhos relativos — consertem os import: uma página em pages/ que usa um componente passa a importá-lo de ../components/; o App.jsx passa a buscar as páginas em ./pages/. E a regra de ouro: não mudem o que cada componente faz — mudem só onde ele mora e como é importado.

Deu certo se…

…a loja continua montando exatamente igual à etapa 2, agora com os 30 arquivos dentro de src/components/ e src/pages/.

04Dividido

Protejam os 25 componentes com testes

Para cada componente, criem um arquivo de teste ao lado dele (co-localizado): CardProduto.jsx ganha um CardProduto.test.jsx na mesma pasta. Todo componente cai em um de três tipos — o comentário no topo do arquivo diz qual — e o tipo diz o que o teste precisa provar:

Tipo 1 · Só aparece na tela (13 componentes)

Mostra conteúdo fixo ou vindo de props, sem interação. O teste prova que o conteúdo está na tela: render + uma busca (getByText, getByRole, getByAltText ou getByLabelText).

Cabecalho · Rodape · Logo · SeloFreteGratis · PrecoEtiqueta · CardProduto · Avaliacao · TagCategoria · Breadcrumb · BadgeEstoque · BannerPromocional · SeloDesconto · IndicadorParcelas

Tipo 2 · Reage ao usuário (7 componentes)

Tem estado interno: a tela muda quando o usuário clica. O teste prova a mudança: renderiza, age com userEvent (sempre com await) e compara a tela antes e depois da ação.

SeletorQuantidade · Acordeao · Abas · GaleriaImagens · ToggleFavorito · BotaoVerMais · MenuCompartilhar

Tipo 3 · Avisa o componente pai (5 componentes)

Recebe uma função por prop (callback) e a chama quando o usuário age. O teste prova o aviso: passa um espião (jest.fn) na prop, age com userEvent e confirma a chamada com toHaveBeenCalledTimes / toHaveBeenCalledWith.

BotaoAdicionarCarrinho · FormularioNewsletter · FiltroPreco · BarraBusca · EstrelasInterativas

Divisão entre as 3 pessoas — mix dos 3 tipos
Pessoa 1 · 9 Cabecalho (1) · Rodape (1) · Logo (1) · SeloFreteGratis (1) · SeloDesconto (1) · SeletorQuantidade (2) · Acordeao (2) · BotaoVerMais (2) · BotaoAdicionarCarrinho (3)
Pessoa 2 · 8 PrecoEtiqueta (1) · CardProduto (1) · Avaliacao (1) · TagCategoria (1) · Abas (2) · GaleriaImagens (2) · MenuCompartilhar (2) · FormularioNewsletter (3)
Pessoa 3 · 8 Breadcrumb (1) · BadgeEstoque (1) · BannerPromocional (1) · IndicadorParcelas (1) · ToggleFavorito (2) · FiltroPreco (3) · BarraBusca (3) · EstrelasInterativas (3)

Comecem cada lote pelos componentes de Tipo 1 (mais simples) e subam para os Tipos 2 e 3. Para rodar só o seu teste enquanto escreve: npm test -- CardProduto (ou npm run test:watch).

Deu certo se…

…há 20 arquivos Componente.test.jsx co-localizados e npm test termina com Test Suites: 25 passed.

¶
Dois componentes usam <Link> do React Router (Cabecalho e Breadcrumb): no teste deles, envolvam o componente em <MemoryRouter> na hora do render. E a 1ª linha de todo teste é import '@testing-library/jest-dom'.

Glossário

rota / React Router
O que decide qual página aparece conforme o endereço na barra do navegador.
export default × export nomeado
Default importa sem chaves: import HomePage from "./HomePage". Nomeado importa com chaves: import { Cabecalho } from "./Cabecalho". Errar as chaves é a causa clássica de componente undefined.
co-localizado
O arquivo de teste fica na mesma pasta do componente que ele testa.
props
Informação (incluindo funções/callbacks) passada de um componente para outro.
userEvent
Simula o usuário de verdade no teste: clique, digitação, tecla. Toda ação dele pede await.
MemoryRouter
Um "router de mentira" para envolver, no teste, componentes que usam <Link>.
json-server
Ferramenta que serve o db.json como uma API de mentira em localhost.
jest.fn
Cria uma função-espião que anota se foi chamada, quantas vezes e com quais valores.

Antes de entregar

Checklist de conclusão

Os mesmos três itens da seção 2 — confiram um a um antes de entregar:

 

1 · A loja monta e navega

npm run dev mostra a loja e as 5 rotas navegam sem erro no console.

 

2 · Projeto organizado

25 componentes em src/components/, 5 páginas em src/pages/, imports apontando certo.

 

3 · Tudo testado e verde

Cada componente tem seu .test.jsx co-localizado, e npm test passa sem falhas.

Selo de conclusão

Se o terminal mostrar Test Suites: 25 passed e a loja navegar no navegador, a casa está arrumada e protegida — vocês concluíram a S.A.
